============================================================
1. 全体構成
============================================================

Trade MVPは、Google Apps Scriptを中心に構築するスマホ向けWebアプリとする。

構成:

スマホブラウザ
  ↓
Google Apps Script Web App
  ↓
Google Drive
  ↓
Googleスプレッドシート

役割:

- スマホブラウザ:
  妻が操作する画面を表示する

- Google Apps Script Web App:
  UI配信、画像保存、入力内容保存、スプレッドシート更新、履歴取得を担当する

- Google Drive:
  エントリー画像と結果画像を保存する

- Googleスプレッドシート:
  trade_id、ステータス、日時、画像URL、ファイルID、メモ、指し値、逆指し値、到達予想時間などを保存する

4. 入力UI仕様
============================================================

指し値入力:
- input type="text"
- inputmode="decimal"
- autocomplete="off"
- maxlength="32"
- placeholder="154.250"
- 保存先: limit_value

逆指し値入力:
- input type="text"
- inputmode="decimal"
- autocomplete="off"
- maxlength="32"
- placeholder="154.000"
- 保存先: stop_value

到達予想時間入力:
- input type="text"
- inputmode="text"
- autocomplete="off"
- maxlength="32"
- placeholder="30m / 1h / 21:30"
- 保存先: expected_reach_time

メモ入力:
- textarea
- maxlength="1000"
- placeholder="ここにメモ"
- 保存先: memo

補足:
- 指し値、逆指し値はtype="number"にしない
- 理由は、154.000のような末尾ゼロや、TP154.250のような文字列をそのまま保存したいため
- 到達予想時間もtype="time"にしない
- 理由は、30m、1h、NYなどの自由な英数字表現を許可したいため
- 3項目はスプレッドシート上でもプレーンテキストとして保存する

   - クライアント側で日付グルーピングしてもよい

8. getTradeById(tradeId)
   - 指定trade_idの記録を1件返す

6. クライアント・サーバー通信
============================================================

通信方法:
- google.script.run を使用する

理由:
- Apps Script HtmlService内でクライアントJavaScriptからサーバー側Apps Script関数を呼び出せる
- 外部APIサーバーを用意しなくてよい
- 無料運用しやすい

通信例:

クライアント側:
- 画像をFileReaderでbase64化する
- base64、ファイル名、MIMEタイプをpayloadに入れる
- google.script.runでサーバー関数を呼ぶ
- 指し値、逆指し値、到達予想時間、メモはdebounceして自動保存する

サーバー側:
- base64文字列をBlobに変換する
- DriveAppで画像ファイルを作成する
- SpreadsheetAppで行を追加または更新する
- 入力値を文字列として保存する
- 結果をJSON相当のオブジェクトで返す

想定payload:

createEntryTrade payload:
{
  images: [
    {
      imageBase64: string,
      fileName: string,
      mimeType: string
    }
  ]
}

updateEntryFields payload:
{
  tradeId: string,
  limitValue: string,
  stopValue: string,
  expectedReachTime: string,
  memo: string
}

completeTrade payload:
{
  tradeId: string,
  images: [
    {
      imageBase64: string,
      fileName: string,
      mimeType: string
    }
  ]
}

9. スプレッドシート操作設計
============================================================

基本方針:
- 1トレード1行
- trade_idを主キーとして扱う
- 行番号は毎回trade_id検索で特定する
- ヘッダー名と列番号の対応表を作る
- 直接「9列目がlimit_value」などの固定依存を避ける

行追加:
- createEntryTradeでappendRowする

行更新:
- updateEntryFieldsで以下を更新する
  - limit_value
  - stop_value
  - expected_reach_time
  - memo
  - updated_at
- completeTradeで以下を更新する
  - result_saved_at
  - duration_minutes
  - result_image_url
  - result_image_urls
  - result_image_file_id
  - result_image_file_ids
  - result_image_filename
  - result_image_filenames
  - result_image_count
  - status
  - updated_at

取得:
- getWaitingTradesでstatus = waitingのみ取得
- getHistoryで全件取得
- getTradeByIdでtrade_id一致の1件を取得

並び順:
- entry_saved_atの降順
- 新しい記録を上に表示する

列フォーマット:
- limit_valueはプレーンテキスト
- stop_valueはプレーンテキスト
- expected_reach_timeはプレーンテキスト
- memoは通常テキスト
- entry_saved_at、result_saved_at、updated_atは日時形式
- duration_minutesは数値形式

注意:
- 指し値や逆指し値を数値として保存しない
- 154.000が154に変換されることを避ける
- 21:30が時刻型に変換されることを避ける
- 入力値はアプリ側で文字列として扱う

============================================================

アップロード方式:
- スマホブラウザで画像ファイルを選択
- FileReaderでbase64化
- Apps Scriptへ送信

ファイル名:
- entry_YYYYMMDD_HHMMSS_{trade_id}.jpg
- result_YYYYMMDD_HHMMSS_{trade_id}.jpg

注意:
- MVPでは画像圧縮は必須ではない
- ただしDrive容量節約のため、将来的にブラウザ側圧縮を追加する
- 大きすぎる画像はApps Script制限に引っかかる可能性があるため、MVP時点でも軽い圧縮を入れられるなら入れる

推奨拡張:
- Canvasで長辺1600px程度に縮小
- JPEG品質0.8前後
- ただし初期実装では複雑化するなら後回し


============================================================
11. 入力値処理設計
============================================================

対象:
- limit_value
- stop_value
- expected_reach_time

保存前処理:
- 前後の空白をtrimする
- 全角英数字を許可するかはMVPでは任意
- 基本は半角英数字前提
- 許可文字以外が含まれる場合はエラーを返す
- 最大文字数を超えた場合はエラーを返す
- 空欄は許可する

許可パターン例:
^[A-Za-z0-9.:_\-/+ ]{0,32}$

補足:
- メモにはこのパターンを適用しない
- メモは日本語、改行、記号を許可する
- 指し値、逆指し値、到達予想時間は文字列保存する

クライアント側:
- 入力時に軽くチェックする
- 不正な文字があれば「英数字で入力してください」と表示する
- ただし最終チェックはサーバー側でも行う

サーバー側:
- validateEntryFieldsで必ず検証する
- 不正な場合はスプレッドシートを更新しない
- エラーメッセージをクライアントに返す

13. エラーハンドリング
============================================================

想定エラー:

1. 画像未選択
   - 表示: 画像を選択してください

2. 許可されていないファイル形式
   - 表示: この画像形式は対応していません

3. 画像アップロード失敗
   - 表示: 画像の保存に失敗しました

4. スプレッドシート更新失敗
   - 表示: 記録の保存に失敗しました

5. 入力内容保存失敗
   - 表示: 入力内容の保存に失敗しました

6. 指し値に許可されていない文字が含まれる
   - 表示: 指し値は英数字で入力してください

7. 逆指し値に許可されていない文字が含まれる
   - 表示: 逆指し値は英数字で入力してください

8. 到達予想時間に許可されていない文字が含まれる
   - 表示: 到達予想時間は英数字で入力してください

9. trade_idが見つからない
   - 表示: 対象の記録が見つかりません

10. 結果保存済みの記録に再度結果画像を保存しようとした
    - 表示: この記録はすでに完了しています

11. 通信中に画面を閉じた
    - 次回起動時にスプレッドシート上の状態を正として表示する

対策:
- 保存中はボタンをdisabledにする
- LockServiceで同時更新をある程度防ぐ
- 例外発生時はクライアントに分かりやすいメッセージを返す
- サーバー側で必ずtrade_idとstatusを検証する
- 入力値はサーバー側でも検証する

14. パフォーマンス方針
============================================================

MVP想定件数:
- まずは数百件〜数千件程度までを想定

履歴取得:
- MVPでは全件取得でも許容
- 件数が増えたら最新100件、月別取得、ページングに拡張する

画像表示:
- 履歴画面では小さめサムネイル
- 詳細画面では大きめ画像
- Drive画像の表示速度に依存する

エントリー補足情報の自動保存:
- 入力停止後1000ms程度で保存
- 連続入力時はdebounceする
- 指し値、逆指し値、到達予想時間、メモをまとめて保存する
- 無駄な書き込みを減らす
